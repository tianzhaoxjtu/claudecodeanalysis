交互深潜

ClaudeCode 的上下文工程不是单点功能，而是“输入预处理 + 工具执行协议 + 多层记忆 + 后台 consolidation”组成的运行时系统

这一页不再做宽泛概览，而是顺着真实代码路径解释：一次输入如何在进入模型前被规范化和压缩；工具如何被注册、筛选、调度和回写；memory 为什么会拆成 memdir、session memory、transcript、agent memory 多层；Dream 为什么本质上是一个后台 memory gardener。

本页回答什么问题

  • 用户输入在进入模型前到底会经历哪些重写、扩展和压缩？
  • 工具不是函数表，而是怎样一种执行协议？
  • ClaudeCode 如何在流式输出过程中并发执行工具，又保持 transcript 正确？
  • Memory 为什么被拆成 memdir、session memory、agent memory 和 transcript 多层？
  • 为什么这些 memory 机制更适合被理解成一个 Claude Code Memory OS，而不是几个零散功能？
  • Dream 在系统里到底扮演什么角色，哪些结论是已确认事实，哪些仍是推断？

关键观察：ClaudeCode 的 context engineering 分成“输入级预处理”和“query 级预处理”两层，后者会在每一轮 query 前持续重写上下文窗口，而不是一次性选完上下文后交给模型。

1. 模型交互前预处理机制

这一层真正做的事，是把原始终端输入编译成主循环可消费的结构化 turn。它同时处理控制流、内容归一化、prompt 路由和上下文前置，不是单纯的“把输入发给模型”。

1.1 机制目的

  • 控制输入生命周期：handlePromptSubmit() 决定本次输入是立即执行、进入队列，还是通过 submit-interrupt 打断当前工具回合。
  • 统一输入形态：文本、粘贴文本、粘贴图片、content blocks、远程 bridge 消息都会在 processUserInputBase() 中被归一成内部消息结构。
  • 在模型前做确定性改写：exit 会重写成 /exit，ULTRAPLAN 会重写成 /ultraplan，remote slash command 只有 bridge-safe 时才在本地执行。

1.2 核心调用链

普通输入的主链路是：handlePromptSubmit() → executeUserInput() → processUserInput() → processUserInputBase() → processBashCommand/processSlashCommand/processTextPrompt → executeUserPromptSubmitHooks() → onQuery() → query()。

  • 入口层：handlePromptSubmit.ts 先用 parseReferences() 和 expandPastedTextRefs() 处理粘贴引用，再过滤孤儿图片。
  • 队列层：queryGuard 配合 enqueue() 保证主 REPL 一次只跑一个主 query；若当前工具都声明可取消，新输入会触发 interrupt。
  • 批处理层：executeUserInput() 对 queued commands 逐个调用 processUserInput()，且只有第一条命令携带 attachments 和 pastedContents，后续命令会 skipAttachments。

1.3 输入编译的内部细节

  • 图片路径稳定化：processUserInputBase() 会先 storeImages() 落盘，再并行 resize 图片，并把尺寸与路径写成 isMeta 消息。
  • bridge 安全边界：remote bridge 输入默认不会触发本地 slash commands，只有 isBridgeSafeCommand() 允许的命令例外。
  • 显式 prompt routing：hasUltraplanKeyword() 命中后立即改写成 /ultraplan，而不是让模型自行推断。
  • 三路 lowering：bash 模式直接调用本地工具且 shouldQuery: false；slash command 再按 local-jsx / local / prompt 分派；普通文本走 processTextPrompt()。

1.4 hooks 和附件如何进入模型上下文

getAttachmentMessages() 负责把文件、目录、IDE selection、memory、task、规则文件、agent/skill 线索整理成 attachment messages。之后 executeUserPromptSubmitHooks() 还可以阻断 continuation、附加 hook_additional_context，甚至用 warning message 替换原始 prompt。

1.5 query-time packing

  • 窗口裁剪：getMessagesAfterCompactBoundary()、applyToolResultBudget()、snipCompactIfNeeded()、microcompact() 共同控制窗口大小。
  • 投影与折叠：contextCollapse.applyCollapsesIfNeeded() 在 autocompact 之前运行，尽量保留比 summary 更细的上下文粒度。
  • 稳定前缀：fetchSystemPromptParts()、getUserContext()、getSystemContext() 构造 cache-critical prefix，再由 prependUserContext() 与 appendSystemContext() 进入最终请求。

1.6 错误处理、回退与工程取舍

  • 尽早短路：未知或不允许的 remote slash command 会直接返回本地提示，不把危险或无效命令继续泄漏给模型。
  • 错误语义修正：processBashCommand() 会把 shell 中断识别成 user interruption，而不是普通工具错误。
  • hook 限流：executeUserPromptSubmitHooks() 的输出会经过截断处理，避免 hook 自己把上下文预算打爆。
  • 组合复杂度：这一层的弱点不是单点 bug，而是旗标组合很多，像 skipSlashCommands、bridgeOrigin、isMeta、skipAttachments 这些状态需要跨多个函数一起理解。

关键观察：真正送到主模型的已经不是用户原始输入，而是经历了引用展开、附件注入、hook 改写和窗口压缩之后的活动上下文。

工程判断：ClaudeCode 把“能确定性完成的上下文工程”前移到代码侧，把真正需要推理的部分留给模型。这样做的收益是更稳的 prompt cache、更少的上下文噪声，以及更明确的安全边界。

2. 工具体系与调度编排

ClaudeCode 的工具系统本质上是一套协议栈，而不是“模型调用本地函数”。工具定义、暴露范围、执行顺序、权限决策、hook 与 transcript 回写都在这一层完成。

2.1 Tool 抽象为什么是胖接口

  • 协议定义：src/Tool.ts 中的 Tool 至少包含 inputSchema、validateInput、isConcurrencySafe、interruptBehavior、backfillObservableInput、mapToolResultToToolResultBlockParam 等切面。
  • 运行时能力对象：ToolUseContext 不只是执行参数，还携带 messages、abortController、toolDecisions、readFileState、getAppState() 等整轮 query 的状态。

2.2 工具如何注册和筛选

  • 基础工具池：getAllBaseTools() 收集 built-in tools，并按 feature gate 决定是否引入某些工具，例如 ToolSearch。
  • 可见工具池：getTools(permissionContext) 再根据 mode、deny rules、REPL 隐藏规则和 isEnabled() 做二次过滤。
  • 最终暴露面：assembleToolPool() 将 built-in 与 MCP tools 合并，形成模型真正可见的工具集合。
  • 隐藏约束：deferred tool 的 schema 不一定真的发给模型；buildSchemaNotSentHint() 会在失败时提示先调用 ToolSearch。

2.3 单个工具调用的真实执行链

  • 入口：runToolUse() 先查当前可见工具池，不命中时再尝试 alias fallback。
  • schema 与业务校验：checkPermissionsAndCallTool() 先做 inputSchema.safeParse()，再做 tool.validateInput()。
  • 输入改写：backfillObservableInput() 只在可观测副本上补字段，让 hooks 和 permissions 可见，但尽量不污染传给 tool.call() 的真实输入。
  • 执行协议：runPreToolUseHooks() → resolveHookPermissionDecision() → tool.call() → runPostToolUseHooks() / runPostToolUseFailureHooks()。

文件映射：src/tools.ts、src/Tool.ts、src/query.ts、src/services/tools/toolOrchestration.ts、src/services/tools/StreamingToolExecutor.ts、src/services/tools/toolExecution.ts。

关键观察：主模型负责发出 tool_use，runtime 负责让这些 tool_use 在统一协议下执行、回写和恢复。

2.4 调度与并发语义

  • 普通批处理：partitionToolCalls() 按 tool.isConcurrencySafe(parsedInput) 分并发批和串行批。
  • contextModifier 乱序保护：并发批里的工具可同时执行，但 contextModifier 会在批次结束后按原始 tool 顺序应用。
  • 流式执行：StreamingToolExecutor 在 assistant streaming 阶段边收到 tool_use 边执行，并维护 queued / executing / completed / yielded 状态。
  • 失败隔离：只有 Bash 错误会级联取消兄弟工具；普通读取类工具失败不会导致整个 batch 取消。

这套设计的优势是执行链统一、hook 和权限可以无缝插入、telemetry 可观测性强。代价是 Tool 抽象相当厚，工具作者必须理解 schema、permission、hook、mapping、interrupt、streaming 等多个维度。

3. Memory 记忆机制

这一层不能简单理解成“严格三层 memory”。从代码看，ClaudeCode 更接近两套正交分类：一套是会被装入上下文的 file-based memory source，另一套是 session summary、agent memory、transcript、运行时缓存等连续性机制。源码里并没有一个官方的统一 layer count。

3.1 memory 是怎么启用和初始化的

  • 总开关：isAutoMemoryEnabled() 位于 src/memdir/paths.ts，优先级链依次检查环境变量、--bare、remote 持久化能力、settings.json 的 autoMemoryEnabled，最后默认开启。
  • 子机制开关：extract-memories、session memory、autoDream 都有自己的 gate。它们不是简单继承总开关，而是再叠加 GrowthBook 和运行模式限制。
  • 初始化入口：loadMemoryPrompt()、initSessionMemory()、initExtractMemories()、initAutoDream()、loadAgentMemoryPrompt() 分别把 memory 接入 system prompt、post-sampling hook、turn-end hook、background housekeeping 和 agent loader。

3.2 路径、目录和安全边界

  • auto memory 目录：getAutoMemPath() 支持 env override、可信 settings override，以及默认的 ~/.claude/projects/<sanitized-git-root>/memory/。
  • 路径安全：validateMemoryPath() 会拒绝相对路径、根路径、UNC 路径、空字节路径，并故意排除 projectSettings 对 autoMemoryDirectory 的控制，防止恶意仓库把 memory 指到敏感目录。
  • 权限 carve-out：filesystem.ts 会对 isAutoMemPath()、isAgentMemoryPath() 提供专门的读写放行，但当使用 CLAUDE_COWORK_MEMORY_PATH_OVERRIDE 时，auto memory 的静默写入 carve-out 会失效，重新走常规 permission flow。

3.3 存储模型与表示方式

  • 显式 source taxonomy：src/utils/memory/types.ts 定义了 Managed / User / Project / Local / AutoMem，在 TEAMMEM 打开时再加上 TeamMem。这是代码里最明确的 memory source 类型系统。
  • auto memory：Markdown 文件加 frontmatter，type 被约束为 user / feedback / project / reference。这里的 type 是 durable memory 条目分类，不是 Managed / User / Project / Local / AutoMem / TeamMem 那套 source taxonomy。MEMORY.md 只是索引，不是正文。
  • session memory：固定结构的单个 markdown 文件，默认模板在 src/services/SessionMemory/prompts.ts，并支持用户自定义模板与 prompt。
  • transcript：sessionStorage.ts 使用 JSONL 记录会话消息、compact boundary、content replacement、collapse snapshot 等恢复元数据。
  • agent memory：getAgentMemoryDir() 支持 user / project / local 三种 scope，并可通过 snapshot 机制同步到本地副本。
  • execution cache：readFileState、contentReplacementState、loadedNestedMemoryPaths 用于去重、预算和 freshness，不是语义记忆。

3.4 memory 是怎么被读出来的

  • 稳定前缀读取：getUserContext() 会通过 getMemoryFiles() / getClaudeMds() 组装 memory。真实装载顺序是 Managed -> User -> Project -> Local -> AutoMem -> TeamMem；其中 project/local 会按目录向上遍历，越靠近当前目录优先级越高。
  • query-time 检索：startRelevantMemoryPrefetch() 会异步启动 findRelevantMemories()，后者先 scanMemoryFiles() 再 sideQuery() 选择候选，最后以 relevant_memories attachment 形式注入当前 turn。
  • nested memory：memoryFilesToAttachments() 会把特定文件相关的 memory / rules 变成 nested_memory 附件，并用 loadedNestedMemoryPaths + readFileState 双重去重。
  • session memory 消费：getSessionMemoryContent() 会被 sessionMemoryCompact.ts、awaySummary.ts、skillify.ts 读取，用于压缩、recap 和技能生成。

3.5 memory 是怎么被写回去的

  • 主代理直接写：system prompt 已告诉主模型如何写 memory file 和更新 MEMORY.md。若主代理本 turn 已写过 memory，extractMemories.ts 会通过 hasMemoryWritesSince() 检测到并跳过背景提取。
  • extract-memories：runExtraction() 会在 turn 结束后扫描新增消息区间，把现有 memory manifest 预注入提示词，再用 runForkedAgent() 在 querySource='extract_memories' 下更新 durable memory。
  • session memory：extractSessionMemory 在主 REPL 的 post-sampling hook 中触发，先创建或读取 session memory 文件，再用受限 subagent 编辑这一条确切路径。
  • agent memory：agent 定义里如果声明了 memory scope，loadAgentsDir.ts 会自动注入 Read/Edit/Write 工具，并在 system prompt 中追加专属 memory 使用说明。

3.6 调度、生命周期与一致性

  • session memory 调度：使用 post-sampling hook + 阈值判断 + sequential()，只在 repl_main_thread 运行，并通过 waitForSessionMemoryExtraction() 给 compaction 路径提供超时等待。
  • extract-memories 调度：使用闭包状态维护 lastMemoryMessageUuid、inProgress、pendingContext，支持 coalescing 和 trailing run，避免多个 extraction 相互重叠。
  • Dream 调度：不依赖当前 turn，而是依赖上次 consolidation 时间、最近 session 数、scan throttle 和 lock。
  • 一致性锚点：lastMemoryMessageUuid、lastSummarizedMessageId、readFileState、hasMemoryWritesSince() 共同保证“不会重复摘要、不会重复 surfacing、不会主代理和后台代理双写同一批信息”。

3.7 端到端 memory 流

memory 对后续交互的影响有三条回流路径：MEMORY.md 进入稳定前缀，relevant memories 进入当前 turn attachments，session memory 进入 compaction / away summary / skillify。与此同时，extract-memories 与 Dream 又在后台更新 durable memory，反过来影响未来 turn 与未来 session。

文件映射：src/memdir/paths.ts、src/utils/claudemd.ts、src/utils/attachments.ts、src/services/SessionMemory/sessionMemory.ts、src/services/extractMemories/extractMemories.ts、src/services/autoDream/*、src/utils/sessionStorage.ts。

关键观察：ClaudeCode 的 memory 是运行时基础设施，而不是单点“记忆库”。durable memory、session memory、transcript 和 agent memory 在不同阶段被不同路径消费和更新。

重要设计原则：src/memdir/memoryTypes.ts 明确禁止把代码结构、架构、git 状态、当前 task context 等可重新推导信息直接写入 memory。系统想保存的是“难以重新发现但值得长期保存的事实”，而不是复制当前仓库状态。

3.8 现实校正：实现里的 memory 不是严格三层

如果严格按代码而不是按概念模型来数，ClaudeCode 没有一个官方的“三层 memory stack”。更接近现实的是两套正交分类：一套是 Managed / User / Project / Local / AutoMem / TeamMem 这类 file-based source taxonomy，另一套是 SessionMemory、AgentMemory、transcript、运行时缓存这些 continuity mechanism。

3.8.1 如果只看会被装入上下文的 memory source，有几种

  • 固定 5 种：Managed / User / Project / Local / AutoMem，定义在 src/utils/memory/types.ts 与 src/utils/claudemd.ts。
  • 可选第 6 种：TeamMem，只有 feature('TEAMMEM') 打开时存在。
  • 层级加载顺序：Managed -> User -> Project -> Local -> AutoMem -> TeamMem；其中 project/local 会按目录向上遍历，越靠近当前目录优先级越高。
  • 配置影响：projectSettings/localSettings/userSettings、claudeMdExcludes、@include、additional directories、nested worktree 去重都会改变最终注入集合。

3.8.2 如果看整个 continuity architecture，至少还有哪些机制

  • SessionMemory：summary.md 形式的会话摘要，主要服务于 compact、away summary、skillify。
  • AgentMemory：user / project / local 三种 scope 的 agent 持久记忆，和前面的 file source taxonomy 正交。
  • Transcript：sessionStorage.ts / sessionRestore.ts 的 JSONL 持久化与恢复基座。
  • 运行时状态：messagesForQuery、toolResults、readFileState、loadedNestedMemoryPaths 等执行期结构。

3.8.3 三层模型和实现现实的差距

  • 三层模型的价值：它帮助理解时间尺度，把当前回合、当前 session、跨 session durable knowledge 分开看。
  • 实现现实的差距：Managed/User/Project/Local 这套 CLAUDE.md/rules 层级不自然落在“三层”里；它更像上下文指令装载系统。
  • 另一个差距：AgentMemory(user/project/local) 本身又是一套 scope taxonomy，不等于 project/session/working 三分法。
  • 结论：三层模型应被视为阅读辅助抽象，而不是源码里的官方架构。

3.9 Claude Code 的 Memory：分析视角与实现现实

更贴近实现的表达不是“源码内置的官方三层 memory 架构”，而是两套系统：一套是文件型、层级化、可持久化的 memory source system；另一套是运行时 continuity system。把它们统称为 Memory OS 是分析视角，用来强调这不是单点存储，而是上下文装配与连续性维护机制的组合。

3.9.5 为什么仍可把它视为 Memory OS

  • 从职责组合看：它同时负责层级文件装载、上下文装配、连续性维护、后台整理和访问治理，已经超出普通“记忆存储”的范围。
  • 从实现现实看：更贴近代码的模型是“文件型 source system + 运行时 continuity system”，而不是单一三层 stack。
  • 从代码证据看：getMemoryFiles()、SessionMemory、transcript、extractMemories、autoDream、permission carve-out 共同组成了这套 memory runtime。
  • 局限：这是分析视角，不是源码里的官方命名；同时 taxonomy 和 mechanism 都比较多，认知负担不低。

3.10 Claude Code Context Engine 架构与 Memory 在其中的位置

Context Engine 同样不是源码中的官方命名，而是对完整上下文构造流水线的架构概括。它把输入预处理、文件型 memory source、运行时 continuity、附件收集、query 打包、工具循环和 turn 后写回放到同一条主线上，从而解释 ClaudeCode 为什么更像一个“上下文构造系统”，而不仅仅是“带记忆的聊天工具”。

3.10.1 Context Engine 总体架构图

它在说明什么：这张图把完整的上下文构造系统放到一张架构图里，明确 memory 只是 Context Engine 的一部分，而不是独立于主循环的孤立模块。

文件映射：src/utils/handlePromptSubmit.ts、src/utils/processUserInput/processUserInput.ts、src/context.ts、src/utils/claudemd.ts、src/utils/attachments.ts、src/query.ts、src/services/tools/*、src/services/SessionMemory/*、src/utils/sessionStorage.ts。

关键观察：真正把所有输入源汇总成“最终 context window”的地方是 query()；memory 既是上游输入，也是 turn 后写回目标。

3.10.2 Context Construction Pipeline 与交互循环图

它在说明什么：这张图展示 Context Engine 不是一次性 assemble，而是一个持续循环：装配上下文、调用模型、执行工具、更新上下文、写回 continuity，再影响下一轮交互。

文件映射：src/context.ts、src/utils/queryContext.ts、src/utils/attachments.ts、src/query.ts、src/services/tools/*、src/services/SessionMemory/*、src/services/extractMemories/extractMemories.ts、src/services/autoDream/autoDream.ts。

关键观察：context window 的真实内容不是启动时一次选定的，而是在工具执行和压缩过程中被持续重写；memory 在读和写两个方向都参与了这条主回路。

3.10.3 Memory 为什么必须放回 Context Engine 里理解

  • 从系统能力看：ClaudeCode 的关键能力不是“有记忆”，而是“能持续构造可用上下文”。memory、文件读取、诊断、任务、工具结果都只是 Context Engine 的不同输入源。
  • 从 context window 组成看：最终窗口至少包含 system prompt、user context、messagesForQuery、attachments、toolResults，以及 compact/collapse 后保留下来的内容。
  • 从代码现实看：memory 会在 session 启动和 turn 执行中被装载，但最终能不能进入模型，仍然要经过 query.ts 的预算治理与裁剪。
  • 限制：context window 仍是硬约束，所有 memory 和附件都必须在打包和压缩后竞争同一预算。

3.11 Claude Code / Cursor / Copilot 的 Context Engine 架构对比

这一节不是做产品功能罗列，而是把三类 coding assistant 放到统一的 Context Engine 视角下比较：谁负责收集上下文、谁负责记忆注入、谁主导工具循环、谁承担 orchestration。这里的证据边界需要特别说明：Claude Code 结论来自当前仓库源码；Cursor 与 GitHub Copilot 主要依据官方公开行为，再辅以架构推断，因此会显式区分“已确认公开行为”和“合理推断”。

3.11.1 对比架构图

它在说明什么：这张图把三者都当成“上下文构造系统”来看，而不是按产品功能比较。核心问题是谁来负责 assemble context、memory 放在哪里、工具回路与上下文更新耦合多深。

文件映射 / 公开资料映射：Claude Code 对应 src/utils/handlePromptSubmit.ts、src/utils/processUserInput/processUserInput.ts、src/query.ts、src/services/tools/*、src/services/SessionMemory/*、src/services/extractMemories/*、src/services/autoDream/*。Cursor 侧主要依据其官方文档中的 Rules、Memories、codebase indexing/agent 描述；Copilot 侧主要依据 GitHub 官方文档中的 repository custom instructions、AGENTS.md、custom agents、MCP、coding agent 描述。

关键观察：Claude Code 的 Context Engine 最显式，query() 是清晰可见的上下文装配核心；Cursor 更像 IDE 内嵌的检索增强上下文层；Copilot 则更像把 GitHub 资源面、平台 orchestration 和编辑器入口拼接成更平台化的 context engine。

3.11.2 对比 context flow 图

它在说明什么：这张图对比一次典型编码任务里三者如何逐步构造和更新上下文。重点不在“会不会调用工具”，而在工具结果、memory、检索结果是否被显式折返进下一轮 context。

文件映射 / 公开资料映射：Claude Code 的读写回路可直接映射到 src/query.ts、src/services/tools/toolExecution.ts、src/utils/sessionStorage.ts、src/services/extractMemories/extractMemories.ts。Cursor 与 Copilot 的 flow 来自官方公开的 agent / memory / repository instructions / MCP 能力，再结合工程上常见的检索增强回路做保守推断。

关键观察：Claude Code 的上下文更新回路最“显式可追踪”；Cursor 的关键差异在于 index/retrieval 先行；Copilot 的关键差异在于上下文构造更可能依赖 GitHub 平台侧资源和远端 orchestration。

3.11.3 逐维度比较

  • Context construction：Claude Code 的已确认实现是把 system prompt、user context、memory、attachments、tool results 和 compact 策略集中到 query()。Cursor 的已确认公开行为表明其强依赖代码库索引与 IDE 当前上下文；Copilot 的已确认公开行为表明其会消费 repo instructions、agent task metadata 和 GitHub 资源面。合理推断是：后两者的上下文构造更依赖宿主平台提供的隐式上下文，而不像 Claude Code 这样在单个 runtime 核心里显式拼装。
  • Memory integration：Claude Code 的 memory 明显分成“文件型 source taxonomy”和“runtime continuity mechanisms”两套正交系统，并通过 context.ts、claudemd.ts、transcript、SessionMemory 汇入主循环。Cursor 的 Rules/Memories 更接近 IDE 宿主提供的持久规则与偏好层；Copilot 更偏向 repository instructions 与 task/container state。对比下来，Claude Code 的 memory 更像 context engine 的可见组件，另外两者更像嵌入平台的 context substrate。
  • Tool interaction：Claude Code 是已确认的 tool-first agent loop，模型产生 tool_use 后进入 StreamingToolExecutor 与 toolExecution，工具结果再折返进消息流。Cursor 与 Copilot 的公开行为都显示它们支持 read/search/edit/terminal 或 agent actions，但内部是否采用与 Claude Code 一样显式的 tool protocol 和消息折返结构，公开证据不足，只能保守推断为“存在类似回路，但封装层更深”。
  • Runtime orchestration：Claude Code 的 orchestration 可以直接在源码里看到：前处理、context assembly、tool loop、writeback、background memory maintenance 是一条连续 runtime。Cursor 的 agent orchestration 很可能深度嵌入 IDE；Copilot 的 coding agent 与 GitHub 任务执行表明其 orchestration 更偏平台侧、多入口与远端协调。换句话说，Claude Code 强在“运行时可解释性”，Cursor/Copilot 更可能强在“宿主整合深度”。
  • Repository understanding：Claude Code 主要依赖搜索、文件读取、repository detection、memory files 和工具循环逐轮构造理解，而不是预设统一代码图谱。Cursor 的公开行为显示其更强调 codebase indexing 和相关代码召回；Copilot 则更可能结合 repo metadata、GitHub graph、当前编辑器上下文与任务对象。对于大仓库，Claude Code 更灵活但更依赖多轮探索；Cursor/Copilot 理论上能更快召回广域结构，但代价是可解释性降低、隐藏策略更多。
  • 工程 trade-off：Claude Code 的优势是上下文主循环清晰、memory 与工具编排能被工程师直接理解和修改；限制是显式工具回路和上下文预算治理会带来更多运行时分支。Cursor 的优势大概率是索引增强和 IDE 原位体验；限制是核心 context engine 对外透明度较低。Copilot 的优势大概率是 GitHub 平台资源整合和远程 agent 能力；限制是平台依赖更强、本地 runtime 边界更不透明。

3.11.4 架构判断

判断：从 Context Engine 架构看，Claude Code 更接近“显式 runtime context machine”；Cursor 更接近“IDE 内嵌的检索增强 context layer”；GitHub Copilot 更接近“平台化 context substrate + agent orchestration”。这不是优劣排序，而是三种工程路径：本地 runtime 显式化、IDE 宿主检索化、平台资源编排化。

3.12 Claude Code Context Engine 五层模型：从实现现实到分析框架

这一节把前面的分析重新组织成一个更系统的五层模型。但要先说清楚：这不是源码里显式命名的官方分层，而是基于当前实现现实提炼出来的分析框架。它的价值在于帮助我们把“输入塑形、持久知识、运行时连续性、检索工具、最终窗口装配”放到同一张地图里理解。

3.12.1 五层模型是什么

  • Layer 1. Input & Intent：用户输入、任务 framing、命令/skill 路由、初始指令塑形。
  • Layer 2. Persistent Knowledge：文件型持久知识，包括 CLAUDE.md、nested memory、SKILL.md、AutoMem、TeamMem 等。
  • Layer 3. Runtime Continuity：transcript、session memory、agent memory、runtime cache、compact boundary 等运行期连续性状态。
  • Layer 4. Retrieval & Tooling：attachments、文件读取、搜索、命令执行、编辑工具，以及它们带回的新鲜仓库事实。
  • Layer 5. Active Context Assembly：把前四层压缩成最终模型上下文窗口，并在 tool loop 中持续重写。
分析框架 vs 实现现实：代码里并没有“五层接口”或“五层目录”，这个模型只是把已经确认的实现机制按职责域重新整理。它是帮助理解的地图，不是源码里的官方类图。

3.12.2 五层静态架构图

它在说明什么：这张图展示的是五个责任域的静态关系。重点不是严格分层调用，而是强调第五层 Active Context Assembly 才是真正的汇合点。

文件映射：第一层对应 src/utils/handlePromptSubmit.ts、src/utils/processUserInput/processUserInput.ts；第二层对应 src/context.ts、src/utils/claudemd.ts、src/memdir/*；第三层对应 src/utils/sessionStorage.ts、src/services/SessionMemory/*、src/tools/AgentTool/agentMemory.ts；第四层对应 src/utils/attachments.ts、src/tools/*、src/services/tools/*；第五层对应 src/query.ts、src/utils/queryContext.ts。

关键观察：memory 在这里只是第二、三层的一部分输入；真正决定“模型此刻看到什么”的，不是 memory 本身，而是第五层的组装与裁剪逻辑。

3.12.3 端到端上下文流图

它在说明什么：这张图按一次真实任务的推进方向展示五层如何协作。任务不是“先把上下文找全再调用模型”，而是在运行时不断把 memory、检索结果和工具输出回流到活动窗口。

文件映射：handlePromptSubmit() / processUserInput() 对应第一层，context.ts / claudemd.ts / memdir/* 对应第二层，sessionStorage.ts / SessionMemory / agentMemory 对应第三层，attachments.ts / services/tools/* 对应第四层，query.ts 对应第五层。

关键观察：Claude Code 的项目理解并不主要来自预先构建的统一代码图谱，而是依赖第四层把相关代码逐轮拉入第五层，再通过模型和工具的交替推进扩展理解范围。

3.12.4 交互循环图

它在说明什么：这张图说明 Claude Code 的主循环里，中心节点不是 memory，而是第五层的活动上下文装配。memory、tools、runtime continuity 都围绕它输入与回写。

文件映射：query.ts 负责第五层的 assemble/refresh，services/tools/* 驱动第四层回路，utils/sessionStorage.ts、services/SessionMemory/*、services/extractMemories/*、services/autoDream/* 承担第三层和第二层的写回。

关键观察：所谓“working set”并不是一个独立官方 subsystem，而是第五层在当前时刻维护的活动窗口。它会随着工具结果、compact 和 writeback 边界不断变化。

3.12.5 逐层角色与工程意义

  • 第一层：决定“这次任务到底是什么”，控制命令路由、引用展开、hook 和安全边界。如果第一层出错，后面层级拿到的就不是正确任务。
  • 第二层：提供跨会话持久知识基底，但它不是最终上下文本身，仍要经过第五层裁剪后才能进模型。
  • 第三层：负责长对话和多步任务的连续性，解决 compact、恢复、forked agent、turn 间延续等问题。
  • 第四层：是 repository understanding 的增量引擎。Claude Code 对陌生代码库的理解很大程度上依赖这一层持续把相关事实拉进来。
  • 第五层：是整个 Context Engine 的中心。真正的系统能力不是“保存了多少 memory”，而是“能否持续装配出当前最有用的上下文窗口”。

3.12.6 这个模型为什么有用

  • 帮助校正重心：它把分析重心从 memory 挪回 context construction，避免把 Claude Code 误读成 memory-first architecture。
  • 帮助解释复杂度热点：query.ts 之所以复杂，是因为第五层承担了全系统的汇合责任，而不是单纯“模型调用函数”。
  • 帮助评估 trade-off：这套设计的强项是上下文主循环显式、可审计；难点是预算竞争激烈，很多机制都会把信息压向同一个活动窗口。

4. Dream 机制分析

从可见代码看，Dream 的真实角色是后台 memory consolidation。它的目标不是帮助当前对话“多想一步”，而是主动整理 durable memory，让未来会话更容易恢复上下文。

4.1 触发与门控

  • 入口：backgroundHousekeeping.ts 会调用 initAutoDream()，后者在闭包里维护 lastSessionScanAt，这是测试友好的设计。
  • 门控顺序：autoDream.ts 先检查 isGateOpen()，再读 lastConsolidatedAt，随后检查时间门限、scan throttle、session 数门限，最后才尝试拿锁。
  • 默认阈值：minHours = 24，minSessions = 5，并通过 GrowthBook cached config 或用户设置覆盖。

4.2 锁和失败恢复

  • 锁文件双重语义：.consolidate-lock 的 mtime 既表示“锁存在”，又表示“上次 consolidation 时间”。
  • PID 校验：tryAcquireConsolidationLock() 写入 PID 后还会读回校验，避免两个 reclaiming 进程都误判自己拿到锁。
  • 失败回滚：rollbackConsolidationLock(priorMtime) 会把 mtime 回滚到尝试前的时间，确保失败的 dream 不会整体推迟下一次触发。

4.3 Dream 子代理如何运行

  • 子代理参数：runForkedAgent() 使用 querySource: 'auto_dream'、forkLabel: 'auto_dream'、skipTranscript: true，说明 Dream 是后台维护任务而不是普通 turn。
  • 权限约束：createAutoMemCanUseTool(memoryRoot) 只允许读 transcript / memory，并只允许在 auto-memory 目录内写。
  • 提示词结构：buildConsolidationPrompt() 把工作流拆成 Orient、Gather、Consolidate、Prune 四个阶段，并要求 index 文件保持尺寸上限。

4.4 UI 映射与不确定性

DreamTask.ts 并不实现 dream 逻辑，而是给后台 fork 映射一个可见 task。它只维护 starting / updating 两个 phase，并通过 assistant turn 的 Edit/Write tool_use 收集 filesTouched。源码注释明确承认这只是近似视图，无法捕获 bash 间接写入。

确认与不确定性：自动 Dream 是已确认实现。手动 /dream 在注释层高度可疑，但当前快照里 recordConsolidation() 没有可见调用点，因此不能当成已闭合事实。

5. 技术总结与架构启示

  • 整体架构意图：把主对话循环集中到 query()，把确定性的上下文工程前移到预处理和 runtime gating，把长期维护任务放到受限 subagent 和后台 housekeeping。
  • 最大优势：prompt cache 意识强、工具协议统一、memory 分层清晰、后台 consolidation 能主动维护上下文 substrate。
  • 最大复杂度来源：query.ts 和 ToolUseContext 承载了过多横切关注点，很多行为需要跨 compact、permissions、hooks、tool loop、memory 才能推理完整。
  • 关键 trade-off：系统获得了强 runtime 控制和强恢复能力，但代价是主循环厚重、feature gate 多、学习曲线陡，而且缺少统一 repo graph 时，深层项目理解仍依赖搜索与读取回合。

放在现代 AI coding tool 语境里，这套实现更像一个 runtime context operating system：它持续装配上下文、持续执行外部动作、持续治理窗口、并在后台维护未来会话的记忆 substrate。
